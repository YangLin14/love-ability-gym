\#\# 1\. 模組概述 (Module Overview)

\#\#\# 1.1 目標  
本模組是整個 App 的基石。目標是幫助用戶從「受害者模式」（都是別人的錯）切換到「掌控者模式」（我能管理我的反應），並提供即時工具來降低情緒強度。

\#\#\# 1.2 用戶故事 (User Stories)  
1\.  **\*\*作為一個憤怒的用戶\*\***，我想要一個緊急按鈕，讓我能在想吼叫時冷靜下來。  
2\.  **\*\*作為一個感到委屈的用戶\*\***，我想要釐清這到底是「事實」還是我的「想像」，以免錯怪對方。  
3\.  **\*\*作為一個初學者\*\***，我想要學會分辨我到底是「生氣」還是「嫉妒」，因為我總是搞不清楚自己的感覺。  
4\.  **\*\*作為一個修煉者\*\***，我想要練習「向內找」，不再把快樂的鑰匙交給別人。

\#\# 2\. 功能規格詳解 (Detailed Feature Specifications)

本模組包含五個核心練習工具。

\#\#\# 2.1 練習一：歸因轉向器 (The Attribution Shift)  
**\*\*對應理論：\*\*** 向內找 (Turning Inward)  
**\*\*核心邏輯：\*\*** 區分「誰的事」。

\#\#\#\# 2.1.1 UI 流程  
1\.  **\*\*觸發：\*\*** 用戶點擊「我現在很困擾」。  
2\.  **\*\*輸入：\*\*** 用戶輸入困擾事件（例如：「他不回我訊息」）。  
3\.  **\*\*引導式問答 (Wizard Step 1):\*\***  
    \* **\*\*系統問：\*\*** 「因為這件事，誰產生了情緒？」  
    \* **\*\*選項：\*\*** A. 我 / B. 他 / C. 都有  
    \* **\*\*邏輯：\*\*** 無論選什麼，最終引導至「因為我有情緒，所以這是我需要處理的事」。  
4\.  **\*\*引導式問答 (Wizard Step 2):\*\***  
    \* **\*\*系統問：\*\*** 「在這件事中，我有沒有任何做得不夠好的地方？（即使只有 1%）」  
    \* **\*\*輸入框：\*\*** 用戶輸入（例如：「我沒有提前跟他說這事很急」）。  
5\.  **\*\*引導式問答 (Wizard Step 3):\*\***  
    \* **\*\*系統問：\*\*** 「為了讓結果更好，**\*\*我\*\***現在可以做什麼？」  
    \* **\*\*輸入框：\*\*** 用戶輸入行動計畫。  
6\.  **\*\*結算：\*\*** 生成一張「掌控卡」，顯示用戶的行動計畫。

\#\#\#\# 2.1.2 技術細節  
\* **\*\*數據存儲：\*\*** 存入 \`attribution\_logs\` 表。  
\* **\*\*動畫：\*\*** 頁面切換使用 \`SlideTransition\`，營造「層層深入」的感覺。

### **2.3 練習三：故事粉碎機 (Story Buster)**

**對應理論：** 區分事實與故事 (Fact vs. Story)

**核心邏輯：** 攝像機法則 (The Camera Test)。

#### **2.3.1 UI 交互設計 (Tinder 風格)**

1. **輸入階段：** 用戶輸入一個讓自己痛苦的想法（例如：「他不愛我了」）。  
2. **判斷階段：**  
   * 螢幕顯示卡片：「他不愛我了」。  
   * **提示語：** 「一台錄影機，能錄到『他不愛我』這個畫面嗎？」  
   * **動作：**  
     * **左滑 (是故事/想像):** 顯示碎紙機動畫，將卡片粉碎。  
     * **右滑 (是事實):** 收入「事實庫」。  
3. **糾錯機制：**  
   * 如果用戶將「他不愛我」右滑（認為是事實）。  
   * **系統攔截：** 彈出視窗「⚠️ 等等！錄影機錄不到『愛不愛』，只能錄到『行為』。這是一個推論，不是事實。請重試。」  
4. **重寫階段：**  
   * 系統要求用戶將「故事」重寫為「事實」。  
   * *例:* 「他 11 點回家」 (事實)。

#### **2.3.2 觸覺反饋 (Haptics)**

* **粉碎成功：** 重震動 (Heavy Impact)。  
* **糾錯攔截：** 兩次短震動 (Double Tap)。

### **2.4 練習四：穿越按鈕 (Deactivating Buttons / Time Travel)**

**對應理論：** 處理創傷與內在小孩

**場景：** 處理強度 \> 8 的強烈情緒。

#### **2.4.1 流程設計 (Audio Guide)**

這是一個**全螢幕沉浸式**的功能，不允許其他操作。

1. **第一階段：暫停 (Pause)**  
   * 畫面全黑，只有中心微光。  
   * 語音：「停下來。不要說話，不要行動。只關注你的呼吸。」  
   * **互動：** 用戶按住螢幕中心，感受手機隨著呼吸節奏震動 (4-7-8 頻率)。  
2. **第二階段：回溯 (Trace Back)**  
   * 語音：「這種胸口悶痛的感覺，熟悉嗎？在你 5 歲、10 歲的時候，有沒有感受過？」  
   * **輸入：** 簡單的時間輪盤，選擇「童年」、「青少年」、「前任」。(不需要精確，只需喚起記憶)。  
3. **第三階段：安撫 (Soothe)**  
   * 語音：「那是過去的痛，不是現在這個人要傷害你。告訴那個小孩：『我看見你了，你現在很安全。』」  
   * **互動：** 用戶對著麥克風複述：「我現在很安全。」(利用語音識別確認完成)。

---

### **2.5 練習五：快樂天平 (The Happiness Scale)**

**對應理論：** 放下對錯 (Letting Go)

**場景：** 針對生活瑣事、習慣差異。

#### **2.5.1 UI 交互設計**

1. **天平介面：**  
   \*  
   * 左盤標籤：**「我要爭對錯」** (重量圖示：石頭)。  
   * 右盤標籤：**「我要快樂」** (重量圖示：羽毛/光芒)。  
2. **權衡過程：**  
   * 用戶輸入爭執點（如：「襪子亂丟」）。  
   * 系統提示：「這件事涉及原則（外遇/暴力）嗎？」→ 選「否」。  
   * 系統提示：「那這就是習慣差異。你想把砝碼加在哪一邊？」  
3. **決策動畫：**  
   * 用戶拖曳金幣到右盤（快樂）。  
   * 天平向右傾斜，背景變為柔和的綠色，播放清脆的風鈴聲。  
4. **生成通行證：**  
   * 彈出卡片：「我允許 \[伴侶名\] 襪子亂丟。我選擇今天的快樂。」  
   * 用戶長按蓋章確認。

## **3\. 數據與隱私 (Data & Privacy)**

### **3.1 本地資料庫模型 (Isar Schema)**

Dart

@collection  
class EmotionLog {  
  Id id \= Isar.autoIncrement;  
    
  late String triggerEvent; // 觸發事件  
  late String emotionName;  // 情緒名稱  
  late int intensity;       // 強度 1-10  
  late DateTime createdAt;  
    
  // 故事粉碎機結果  
  String? factVersion;      // 重寫後的事實  
  String? storyVersion;     // 原始的故事  
    
  // 歸因結果  
  String? myResponsibility; // 我能負責的部分  
  String? actionPlan;       // 我的行動  
}

### **3.2 隱私策略**

* **加密：** 所有 triggerEvent 和 actionPlan 欄位在寫入 Isar 前進行 AES 加密。  
* **生物鎖：** 進入 Module 1 的歷史記錄頁面時，強制調用 local\_auth (FaceID)。

## **4\. 異常處理 (Edge Cases)**

1. **用戶情緒崩潰 (強度選 10)：**  
   * 系統不應讓用戶繼續填寫「歸因轉向」（那時候聽不進道理）。  
   * **處理：** 強制跳轉至「練習四：穿越按鈕」進行呼吸急救。  
2. **用戶拒絕承認是故事：**  
   * 在「故事粉碎機」中，若用戶連續 3 次將顯而易見的「故事」歸類為「事實」。  
   * **處理：** 暫停練習，提示：「你看起來很受傷。現在可能不是分辨事實的好時機。我們先去休息一下？」(跳轉至白噪音頁面)。
